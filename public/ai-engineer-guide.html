🚀 From Models to Systems: The Complete AI Engineering Playbook

Before we dive into individual concepts, here is the complete picture of how all these layers connect — from raw data ingestion all the way to production monitoring. Every term in this guide maps to one of these layers.

The full AI engineering stack: Data Ingestion → Feature Engineering → Model Usage → Evaluation → Deployment → Monitoring

1The 4-Layer AI Application Stack

Every AI application — from ChatGPT to FinRAG Engine to Perplexity — is built on four distinct layers. This mental model lets you look at any AI product and understand how it works.

🔑 Key insight
AI engineers are application builders, not model trainers. You use existing AI capabilities as building blocks to create products. This is why the skill bar is achievable — you do not need to understand model internals, just how to use them effectively.
THE 4-LAYER AI APPLICATION STACK
LAYER 4: APPLICATION LAYER
User interface, APIs, integrations — what users actually interact with
LAYER 3: ORCHESTRATION LAYER
LangChain, Llama Index — application logic tying components together
LAYER 2: RETRIEVAL LAYER
Embeddings, Vector Databases, Search — giving AI access to specific knowledge
LAYER 1: MODEL LAYER
OpenAI, Anthropic (proprietary) | Fireworks AI, Groq (open-source)

2Foundation — Models, LLMs & APIs
2.1 Large Language Models (LLMs)

An AI model trained on vast amounts of text data that can understand and generate human language. Trained to predict the next token (word piece) given previous context — through this simple objective at massive scale, they learn language, reasoning, facts, and code.

2.2 Proprietary vs Open-Source Models
TypeExamplesProsConsBest For
ProprietaryGPT-4, Claude 3Highest quality, managed infra, no GPUCost per token, data to 3rd partyHigh-stakes tasks, prototyping
Open-SourceLlama 3, Mistral, GemmaData privacy, no per-token cost, customizableRequires infra, lower qualityPrivate data, high-volume, fine-tuning
2.3 Tokenization
What is a token?
~4 characters or ~0.75 words. The basic unit of text LLMs process. You are billed per token.
100 tokens ≈ 75 words · 1,000 tokens ≈ 750 words
Why it matters
Cost, context limits, and quality all depend on tokens. A prompt using 2,000 tokens when 500 would work is 4× more expensive.
Rule of thumb
~1 token ≈ 4 characters in English. Code and special characters use more tokens per character.
2.4 Context Window

The maximum amount of text (in tokens) a model can consider at one time — its "working memory." Everything in the context window influences the response.

ModelContext WindowApprox. Word Count
GPT-4128K tokens~96,000 words
Claude 3200K tokens~150,000 words
Llama 3128K tokens~96,000 words
Older models4K–32K tokens~3,000–24,000 words
⚠️ The "Lost in the Middle" problem
Models may miss information in the middle of very long contexts. RAG is one solution — retrieve only the most relevant content rather than loading everything.

3Prompt Engineering

The discipline of designing, optimizing, and systematically improving the text instructions given to AI models to reliably produce desired outputs. Prompting is engineering — not guessing.

3.1 System Prompt

A special prompt that sets context, persona, rules, and behavior for the entire conversation. Sent before any user message.

System prompt example
You are a financial analyst assistant.
- Answer only financial questions
- Format all numbers with commas
- Always cite the source of your data
- Do not provide investment advice
- Respond in structured JSON when asked for data
3.2 Core Prompting Techniques
TechniqueWhat It IsWhen to UseExample
Clear InstructionsExplicit, specific direction instead of vague requestsAlways — default approach"Summarize into exactly 3 bullets. Focus on key decisions. Plain English."
Few-Shot2–5 examples of exact input-output behavior before your actual inputConsistent format tasks, data extractionShow 3 examples of the exact output format you want
Chain-of-ThoughtInstruct model to reason step by step before final answerMath, logic, multi-step problemsAdd "Reason through this carefully before answering."
Zero-ShotNo examples given — rely on model's pre-trained knowledgeGeneral tasks, well-defined instructionsJust the instruction, no examples
3.3 Structured Outputs

Getting a model to return data in a machine-readable format (usually JSON) rather than free-form text. Never trust that the model returned valid JSON — always parse and validate.

Structured output prompt
Extract the following from the email and return as JSON only:
{
  "sender": string,
  "subject": string,
  "action_items": list[string],
  "priority": "high" | "medium" | "low"
}

Do not include any text outside the JSON object.
3.4 Rate Limits & Cost Management
ConceptWhat It IsEngineering Response
Rate LimitsMax requests per minute (RPM) and tokens per minute (TPM)Implement retry with exponential backoff: 1s → 2s → 4s → 8s
Cost at scale2,000 tokens when 500 works = 4× cost multiplierOptimize prompt length, use cheaper models when quality allows
CachingStore LLM responses for reuse on identical queriesRedis for exact match, GPTCache for semantic similarity
MonitoringTrack cost per request, per user, per daySet budget alerts before they become budget crises
4Building AI Applications
4.1 Tool Use / Function Calling

A capability allowing LLMs to call predefined functions when needed — enabling real actions rather than just generating text. LLMs on their own can only generate text. Tool use bridges the gap.

StepWhat Happens
1You define available tools with JSON schemas (name, description, parameters)
2Model decides a tool is needed and generates a JSON call with parameters
3Your code executes the actual function (search, email, database query, etc.)
4Result is returned to the model as a tool result
5Model incorporates the result into its response to the user

Example tools: search_web(query), get_weather(location), send_email(to, subject, body), query_database(sql), get_stock_price(ticker)

4.2 Chaining

Building AI applications where the output of one LLM call becomes the input of the next — creating a pipeline of AI operations that work together.

5RAG — Retrieval-Augmented Generation

A technique combining information retrieval with text generation. Instead of relying on training knowledge alone, RAG retrieves relevant documents from a knowledge base at query time and provides them as context.

⚠️ The most important RAG insight
Most RAG systems fail because retrieval is weak, not because the LLM is bad. You feed the model irrelevant context and the output suffers no matter how smart the model is. The retrieval quality is the bottleneck.
5.1 The Complete RAG Pipeline
PHASE 1: INDEXING PIPELINE (Done Once)
Raw Documents (PDFs, docs, web)
↓ Chunking → ↓ Embedding → ↓ Vector Database → Ready for Retrieval
PHASE 2: QUERY PIPELINE (Per User Question)
User Query
↓ Query Embedding → ↓ Similarity Search → ↓ Retrieved Chunks → ↓ LLM Generation with Context
5.2 Chunking Strategy Comparison
StrategyHow It WorksProsConsUse When
Fixed-sizeSplit every N tokens (e.g., 512)Simple, predictable, fastMay cut sentences mid-thoughtStarting point for most projects
SemanticSplit at paragraph/sentence boundariesPreserves meaningVariable chunk sizesImportant for accuracy
RecursiveParagraph → sentence → word → characterBest of both worldsSlightly more complexProduction RAG systems
Structure-preservingRespect code blocks, tables, markdownKeeps structure intactRequires parsingTechnical docs, code repos

Chunk overlap: Include 20–50 tokens from the previous chunk in each new chunk to avoid losing context at boundaries. (FinRAG uses 150-character overlap.)

5.3 Embeddings

Numerical representations of text as high-dimensional vectors. Texts with similar meaning produce vectors that are close together in space.

ModelDimensionsCost / 1M tokensNotes
text-embedding-3-small1,536$0.02Best value. Used in FinRAG Engine.
text-embedding-3-large3,072$0.13Higher quality, more storage needed
text-embedding-ada-0021,536$0.10Older, more expensive, lower quality
sentence-transformers768FreeOpen-source, runs locally
🔧 Critical rule
Always use the same embedding model at index time AND query time. Using different models is BUG-16 in FinRAG Engine — it produced completely wrong similarity scores because the vectors lived in different spaces.

5.5 Retrieval Strategies
StrategyHow It WorksWhen to Use
Simple SimilarityEmbed query → find top-k most similar chunksStarting point, single-topic questions
Hybrid SearchVector similarity + BM25 keyword search, merged via Reciprocal Rank FusionQueries needing both semantic AND exact match (e.g., "Return policy for SKU-4821")
RerankingRetrieve top-50, then use cross-encoder to re-score top-5High-stakes applications where quality matters most
Query TransformationLLM rewrites query before retrieval (HyDE, multi-query, step-back)Complex or ambiguous user questions
MultiQueryRetrieverGenerate 3–5 query variations, retrieve for each, merge resultsCompound questions spanning multiple topics

6AI Agents

An AI system that takes multiple steps to accomplish a goal, deciding what actions to take along the way. Unlike a single-turn application, an agent reasons, plans, uses tools, observes results, and iterates.

6.1 The Agent Loop (ReAct Pattern)
👁️
Observe
Current state + goal
🧠
Think
Decide next action
⚡
Act
Execute tool/action
📊
Result
Process outcome
🔄
Repeat
Until goal done
6.2 Planning Techniques
TechniqueHow It WorksCostBest For
Chain of Thought (CoT)Agent writes out reasoning before acting. Linear path.Low — one pathMost tasks. Start here.
Tree of Thought (ToT)Agent explores 3–5 possible next steps simultaneously, picks best branch.High — many LLM callsComplex problems with multiple valid approaches
6.3 Agent Memory Types
TypeWhere StoredDurationLimitation
Short-TermIn the context windowCurrent task onlyContext window limit — overflow = memory loss
Long-TermVector store / databaseAcross sessionsMust decide what to store and when to retrieve
6.4 Agent Challenges
ChallengeWhy It HappensEngineering Solution
CostEach step = one LLM call. 10-step agent = 10× cost.Use cheaper models for simple steps, limit max steps
LatencySequential LLM calls add up. 10 steps ≈ 30–60 seconds.Stream progress to user, parallelize where possible
UnpredictabilitySame input can produce different paths.temperature=0, max step limits, fallback behaviors
ObservabilityHard to debug without visibility into each step.Log every LLM call, tool call, and reasoning step
💡 KISS Principle for Agents
Simpler agents are usually better. Before building a complex agent, ask yourself: could a simple chain or even a single prompt work? The MultiQueryRetriever in FinRAG Engine is the simplest possible agent — one LLM call to decompose a question — and it dramatically improved answer quality.
7Deployment & Production
🚀 The biggest differentiator
Projects running only on your laptop create zero business value. A deployed system that real users can access and that runs reliably is what matters. Deployment knowledge is what separates hobbyists from engineers.
7.1 Key Production Tools
ToolWhat It IsWhy It Matters
FastAPIPython async web framework for APIsIndustry standard for AI service deployment. Automatic docs, Pydantic validation, async support.
DockerContainerization platformEliminates "works on my machine." Ensures identical behavior dev → staging → production.

LangSmithLangChain tracing platformLogs every LLM call, prompt, response, timing, and token usage.

7.2 Production Best Practices
PracticeRule
Environment variablesNEVER hardcode API keys. Use .env files locally, platform secrets in production.
Health checksExpose GET /health returning 200 OK. Used by load balancers to detect failures.
Retry logicExponential backoff: 1s → 2s → 4s → 8s on rate limit hits (HTTP 429).
Graceful degradationReturn helpful fallback response when APIs fail. Never crash with 500 errors.
Cost monitoringTrack cost per request, per user, per day. Set budget alerts.
CachingSkip LLM calls for identical/similar queries. Redis for exact, GPTCache for semantic.
7.3 Monitoring Metrics
Latency
End-to-end request time. Break down: LLM call time vs retrieval time vs other processing.
Cost tracking
Cost per request, per user per day, per feature. Alert on unexpected spikes.
Error rates
% of requests failing. Types: API timeout, rate limit, model error, parsing error.
Quality monitoring
Use LLM-as-judge to continuously evaluate output quality on production traffic.
8Specializations & Advanced Topics
8.1 Fine-Tuning

Taking a pre-trained base model and training it further on your specific domain data to improve performance on that particular use case.

Use Fine-Tuning WhenDo NOT Fine-Tune When
Need consistent style/format not achievable with promptingGood prompting or RAG would achieve the goal
Have 100K+ examples of high-quality domain-specific dataYou have less than ~1,000 examples
Need to teach specialized vocabularyCost and complexity are not justified
Need lower latency or cost at very large scaleYou are prototyping or early in development

LoRA (Low-Rank Adaptation): The most efficient fine-tuning technique. Adds small trainable matrices to specific layers instead of updating all weights. Requires 10–100× less memory than full fine-tuning.

8.2 LLMOps
PracticeWhat It Means
Prompt versioningTrack prompt changes over time like code. Use git for prompts.
A/B testingTest different models and prompts with real traffic to measure improvement.
Automated evaluationRun evaluation metrics automatically on every code change (CI pipeline).
Drift detectionCatch when model behavior changes over time in production.
Human feedback loopsCollect thumbs up/down on outputs and use to improve.

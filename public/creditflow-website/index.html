01 · Platform Overview

Purpose and operating model

CreditFlow automates commercial credit underwriting through typed contracts, deterministic evidence handling, modular agents, and an analyst review workspace.

The repository centers on a simple principle: every underwriting artifact should move through explicit contracts and named layers. Backend schemas, document ingestion, retrieval, memo assembly, and analyst actions are separated into concrete modules so the runtime can stay inspectable and evolvable.

Platform identity

The codebase is named creditflow-intelligence-platform. It provides a local and pilot-ready underwriting runtime with multi-tenant storage models, deterministic agent orchestration, and a dense analyst workspace.

Primary user journey

A user creates a deal, uploads supporting documents, runs underwriting, retrieves a credit memo, and records analyst review actions. Memo values and review context are designed to surface source evidence directly in the UI.

03 · End-to-End Workflow

Upload to memo review, with named boundaries

The architecture document exposes the runtime flow as four linked journeys: deal intake, upload, underwriting execution, and memo review.

01

Deal intake

POST /api/v1/deals binds request, trace, actor, and tenant context in middleware. DealIntakeService persists the deal under the active tenant and can register structured source documents.

02

Document upload and normalization

POST /api/v1/deals/{deal_id}/documents/uploads stores uploaded bytes through the configured object-store boundary. DocumentIngestionService then normalizes the file into a document record and links it back to the deal.

03

Underwriting run

POST /api/v1/underwriting/runs verifies the tenant-scoped deal, creates workflow identity, executes the deterministic agent graph, and persists memo plus critic outputs through the storage boundary.

04

Memo retrieval and analyst action

GET /api/v1/memos/{memo_id} returns the memo, critic assessment, and persisted analyst actions. POST /api/v1/memos/{memo_id}/analyst-actions records the decision under the authenticated actor and tenant scope.

Key operating principle

Routes do not own normalization, persistence, retrieval, or underwriting logic. Services coordinate the flow; storage adapters and retrieval adapters own backend-specific concerns.

08 · Real Versus Boundary-Only

What is real today, and what is still deferred

The repository is explicit about cloud boundaries, live integrations, and production policy claims that remain outside the locally validated surface.

Real today

  • Local and pilot runtime
  • Tenant-aware persistence and retrieval boundaries
  • Signed session manager and RBAC boundary
  • Multipart upload path and PDF extraction provenance
  • Persistent analyst actions and memo review evidence drill-through
  • Full backend regression surface and local evaluation runners

Boundary-only or deferred

  • Live identity provider integration
  • Live PostgreSQL deployment validation
  • Live provider-backed upload storage
  • Live vector database or hybrid managed search backend
  • Guaranteed managed OCR runtime
  • Governed production underwriting policy logic

Why this distinction matters

The codebase does not blur local validation into unsupported production claims. That makes it a stronger engineering artifact: the runtime surface is real, but the repository still documents where cloud guarantees and governed policy logic have not been fully validated.

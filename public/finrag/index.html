01 · Project Overview

What FinRAG Engine v2 is and who it's for

FinRAG Engine v2 is a document-grounded question answering platform for teams that need fast, explainable answers from uploaded business files. It combines a FastAPI backend, a FAISS-based hybrid retrieval pipeline, and a React frontend for ingestion, monitoring, and query workflows.

The repository is suitable for:

  • Product demos that need visible source citations
  • Internal knowledge assistants over PDFs, Word docs, spreadsheets, and text
  • Multi-tenant document workspaces scoped by user_id
  • Structured CSV/XLSX lookups where exact row retrieval matters

02 · Why This Project Matters

The problem with most chat interfaces in demos

Most chat interfaces fail in demos because they answer confidently without showing evidence. FinRAG Engine v2 is built around the opposite behaviour:

Grounded
Answers stay grounded in uploaded documents
The LLM is instructed to answer only from retrieved context. If the answer isn't in the documents, it says so — it does not hallucinate.
Citations
Every factual answer is backed by citations
Responses include [source: filename.pdf, page_4] inline. The frontend shows expandable source cards with content snippets and chunk IDs.
Immediate
New uploads can be queried without retraining
Ingestion runs in the background and indexes immediately. No model retraining — the vector store is updated and the pipeline cache is invalidated on completion.
Transparent
UI exposes confidence, latency, and source chunks
The query workspace shows staged loading, confidence score, retrieval metadata (queries generated, candidates before/after rerank, latency_ms, request_id), and expandable source cards.
Deterministic
Structured table queries use deterministic row matching
Patient-style CSV/XLSX queries bypass semantic retrieval entirely and use exact row filtering — zero hallucination risk for tabular lookups.
Why this matters for demos

This makes the project useful both as an engineering foundation and as a demo artifact that explains itself clearly to customers, operators, and investors. The UI's transparency — showing exactly which documents were used and with what confidence — is the product signal that distinguishes it from generic chat interfaces.

04 · Chunking Strategy

Why chunking is the most underestimated decision in RAG

Chunking directly determines retrieval quality. Chunks too large lose precision — the right answer is buried in noise. Chunks too small lose context — a sentence without its surrounding paragraph is meaningless. The optimal size is also format-dependent: a dense financial PDF needs larger windows than a structured CSV row.

Format-aware automatic inference

Chunk size is inferred from documents[0].metadata["doc_type"] automatically. Callers pass documents and get correctly-chunked output without specifying sizes. Overrides are available for advanced tuning.

python · src/chunking.py
# Per-format defaults: (chunk_size, chunk_overlap)
_FORMAT_DEFAULTS = {
    "pdf":   (1200, 200),  # dense financial text, larger context window
    "txt":   (1000, 150),  # paragraphs pre-split by loader
    "docx":  (800,  150),  # section-grouped; keep sections coherent
    "csv":   (1500, 0),    # rows are independent; no overlap needed
    "excel": (1500, 0),    # same as csv
}

05 · Embeddings & Vector Store

Converting text to vectors and storing them safely

Each chunk is converted to a 1536-dimensional vector by text-embedding-3-small. Similar meanings produce similar vectors, enabling semantic similarity search. The embedding model used at indexing time must be identical at query time — swapping models requires full re-indexing.

python · src/embeddings.py
EMBED_BATCH_SIZE = 200   # safe batch — under OpenAI's 300k token/req limit
_LOCK_TIMEOUT    = 60    # seconds to wait for FileLock before raising

with filelock.FileLock(lock_path, timeout=_LOCK_TIMEOUT):
    for idx, batch in enumerate(batches, start=1):
        if vectorstore is None:
            vectorstore = FAISS.from_documents(batch, embeddings)
        else:
            vectorstore.add_documents(batch)
        vectorstore.save_local(persist_path)  # checkpoint after every batch

The index is checkpointed after every 200-chunk batch. If the process is interrupted mid-ingestion, the next run resumes safely. The write lock prevents index corruption when multiple concurrent ingestion jobs run simultaneously.

C2 — Race condition fix

Multiple concurrent uploads writing to the same FAISS index files caused corruption. threading.Lock only protects within one process. Fix: filelock.FileLock at the OS level — protects against concurrent processes and separate Docker containers sharing a volume.

SHA256 deduplication

Before saving any uploaded file, its SHA256 hash is computed and checked against vector_store/ingested_files.json. Duplicate uploads within the same user namespace return status: "skipped" immediately — no disk write, no embedding cost, no duplicate chunks in the index.

06 · Hybrid Search Design

Why dense-only retrieval isn't enough for business corpora

Dense vector search (FAISS) captures semantic similarity — finding conceptually related chunks even without keyword overlap. But it struggles with exact matches: specific entity names, ticker symbols, patient IDs, and technical codes. BM25 handles exact matches but misses semantic variation. The hybrid ensemble captures both.

Key decision

Hybrid retrieval is a better fit for mixed business corpora than vector-only retrieval because users ask both semantic questions ("what are the risks?") and exact-match questions ("what is clause 4.2?"). A single retrieval strategy cannot handle both well.

python · src/retriever.py
# BM25 built from FAISS docstore — same documents, different scoring
all_docs  = _extract_docs_from_faiss(vectorstore)
bm25_docs = _filter_docs(all_docs, filter_kwargs)  # tenant isolation
bm25_retriever = BM25Retriever.from_documents(bm25_docs)
bm25_retriever.k = top_k

ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, faiss_retriever],
    weights=[0.3, 0.7],  # BM25 / FAISS via Reciprocal Rank Fusion
)

Multi-tenancy is enforced at both legs: FAISS uses native filter_kwargs={"user_id": uid} metadata filtering; BM25 docs are pre-filtered by user_id before the BM25 index is constructed. Neither leg ever returns another tenant's documents.

07 · FAISS vs Alternatives

Why FAISS, and when to migrate away

OptionLatencyInfrastructureScaleBest for
FAISS ✓~1–5msZero (in-process)~5M/nodeSingle-server, local demos, <1M docs
Pinecone~10–30msFully managed SaaSUnlimitedHorizontal scale, no ops
Weaviate~5–15msSelf-hosted or cloudUnlimitedHybrid search built-in
Chroma~2–10msEmbedded or server~1M vecsPrototyping, local dev
Qdrant~5–15msSelf-hosted or cloudUnlimitedProduction, Rust performance

FAISS was chosen because it runs in-process with zero additional infrastructure. For the target scale (single-node, thousands of documents), FAISS is faster than any managed alternative. Migrating means replacing src/embeddings.py — the rest of the pipeline is vector-store-agnostic through LangChain's abstraction layer.

File-based persistence note

The current implementation persists FAISS artifacts in vector_store/, uploaded files in data/raw/, and write coordination in vector_store/.write.lock. This keeps the stack easy to run locally and easy to demo without adding external infrastructure — but it is not horizontally distributed.

08 · Retrieval Pipeline

Five layers applied in order on every query

1
BM25 + FAISS/MMR via EnsembleRetriever
Sparse keyword retrieval (BM25 weight=0.3) + dense semantic retrieval (FAISS/MMR weight=0.7). Results merged via Reciprocal Rank Fusion. MMR uses lambda_mult=0.6 for relevance-diversity balance — prevents 8 near-identical chunks dominating.
2
_SafeEmbeddingsFilter — re-ranking with safety net
Recomputes cosine similarity between the query embedding and every candidate chunk. Drops chunks below RERANK_THRESHOLD=0.30. Safety net: if ALL candidates would be dropped, top-1 is always returned — the pipeline never delivers empty context to the LLM.
3
MultiQueryRetriever — optional query expansion
Generates MULTI_QUERY_NUM_QUERIES=2 alternative phrasings using a cached gpt-4o-mini singleton. Retrieves for all variants and returns the union. Adaptively disabled for short queries (≤4 words, <30 chars). Disable entirely via USE_MULTI_QUERY=false.
4
SafeQueryWrapper — system boundary input guard
Outermost BaseRetriever. LangChain's create_retrieval_chain may pass {"input": str} instead of a plain string. This wrapper normalises all input — making all inner components immune to LangChain API changes (RCA-5).
5
Compound Query Orchestration
Multi-part questions are split into focused clauses. Each clause retrieves independently using COMPOUND_QUERY_TOP_K=4 chunks. All clause contexts feed one LLM call with a structured COMPOUND_PROMPT, answering each clause in its own ### Heading.
RCA-9 — The threshold bug that killed all retrieval

The original RERANK_THRESHOLD=0.75 dropped every chunk for text-embedding-3-small. This model produces cosine similarities of 0.30–0.65 for relevant content — never approaching 0.75. Every query returned "This information is not available." Fix: lower to 0.30 + safety-net fallback returning top-1 when all candidates fall below threshold.

09 · Generation & Prompts

Building grounded answers from retrieved context

python · src/generator.py — system prompt
SYSTEM_PROMPT = """You are FinRAG, an expert document assistant.

Answer the user's question using ONLY the provided context passages.
Each passage starts with a [File: X | Location: Y] label for citations.

Rules:
1. Be concise and direct.
2. Use only supported facts from the context.
3. Cite every factual claim as [source: <filename>, <location>].
4. If the context does not contain the answer, say exactly:
   This information is not available in the provided documents.
5. Preserve exact figures, names, and terminology from the source.

Context passages:
{context}"""

Rule 4 is the hallucination guard. The DOCUMENT_PROMPT template embeds [File: X | Location: Y] into every passage before it reaches the LLM, so citations always reference real metadata — never hallucinated filenames. For compound queries, a separate COMPOUND_PROMPT instructs the LLM to answer each clause in its own ### heading using only the context attached to that specific clause.

Reliability

All LLM calls use tenacity: 4 attempts, exponential backoff (2s min, 30s max), retrying on any exception. Timeout is 45 seconds. The _get_chat_llm function is cached with @lru_cache to avoid per-call model instantiation. max_tokens is capped at 700 (single Q) or 900 (compound) to control cost and latency.

10 · Agentic Patterns

Where the system makes intelligent routing decisions

Adaptive
Adaptive MultiQuery activation
Detects query complexity before invoking MultiQueryRetriever. Short queries (≤4 words, <30 chars) skip it. Complex queries use 2 LLM-generated variants. Controlled by _is_complex_query() in query.py.
Compound
Multi-intent clause splitting
Queries with 2+ intent starters ("What... and also how...") are split into focused clauses. Each clause retrieves independently — prevents blended retrieval across unrelated topics (RCA-12).
Structured
Deterministic patient table lookup
Structured patient queries ("list all female patients from Chennai") bypass vector retrieval and scan CSV/XLSX rows directly using PatientQueryFilters. Exact row matches; zero hallucination risk for tabular data.
Failsafe
Injection guard + graceful failsafe
Prompt injection patterns are blocked at the QueryRequest validation layer before any retrieval occurs. If retrieval returns no context, the system returns "No relevant context found" — never hallucinating from empty context.

14 · Bug Register

Critical issues found and fixed across 21 phases

Critical — production correctness

C1
Blocking I/O in async route handlers
CPU-bound ingestion (load, chunk, embed, FAISS write) ran synchronously inside async def handlers, blocking the event loop. All concurrent requests queued behind one slow upload.
→ Fix: asyncio.to_thread() for all blocking ops in _run_ingestion()
C2
FAISS race condition — concurrent ingestion corrupted the index
Multiple uploads writing to the same FAISS files simultaneously caused corruption. threading.Lock only protects within one process — not between uvicorn workers or separate containers.
→ Fix: filelock.FileLock at OS level on vector_store/.write.lock (60s timeout)
C3
Cache key missing top_k — wrong pipeline served silently
A query with top_k=4 would load the pipeline built for top_k=8. Wrong chunk count, wrong results — silently, every time.
→ Fix: cache key = f"topk={top_k}|filters={...}|mq={use_multi_query}"
C4
Filename collision — second upload overwrote the first
Two users uploading report.pdf resulted in the second overwriting the first on disk.
→ Fix: UUID-prefixed filenames: {uuid4().hex[:8]}_{original_filename}
C6
File size check after full file read — OOM risk
The size guard ran after the entire file was buffered. A 500MB upload would exhaust RAM before rejection triggered.
→ Fix: 64KB streaming chunks with early reject before buffering completes

RAG pipeline — retrieval correctness

RCA-4
_SafeEmbeddingsFilter was a plain class — reranking bypassed entirely
LangChain's MultiQueryRetriever calls .invoke() on the wrapped retriever. A plain class has no invoke method — __getattr__ delegation skipped the reranking logic. Every query got unranked results.
→ Fix: converted to proper BaseRetriever subclass — .invoke() now routes through _get_relevant_documents()
RCA-5
Missing SafeQueryWrapper — dict input crashed inner retrievers
LangChain's create_retrieval_chain passes {"input": str} in some code paths. BM25, FAISS expect plain str — TypeErrors at retrieval time.
→ Fix: SafeQueryWrapper(BaseRetriever) as mandatory outermost layer; normalises dict → str
RCA-7
MultiQueryRetriever num_queries silently ignored — always used 3
Setting retriever.num_queries = 2 post-construction raises ValueError in Pydantic v2. Assignment was silently dropped. All queries used 3 variants regardless of settings.
→ Fix: custom PromptTemplate passed to from_llm(prompt=...) embedding count in template text
RCA-9
Rerank threshold 0.75 dropped ALL chunks — empty context on every query
text-embedding-3-small produces cosine similarities of 0.30–0.65 for relevant content. The 0.75 threshold was above the model's maximum. Every query returned "This information is not available."
→ Fix: threshold lowered to 0.30 + safety-net returning top-1 when all candidates fall below
RCA-12
Mixed-intent queries retrieved blended context — both topics answered poorly
A compound query ("What is X? And also Y?") caused single retrieval over both topics. Top-k chunks mixed content from both, diluting both answers.
→ Fix: clause splitting → independent retrieval per clause → single compound generation pass
H4
Misleading health check — healthy on empty or corrupt index
GET /health checked directory existence only. Empty vector_store/ returned vectorstore_loaded: true.
→ Fix: faiss.read_index().ntotal — true only when ntotal > 0

17 · Limitations & Next Steps

Honest assessment of current limits and the path to scale

Current limitations

  • pipeline_cache, query_result_cache, and job_registry are in-memory and not shared across multiple backend workers. Running --workers 4 gives 4 independent caches.
  • FAISS is stored on local disk and is not horizontally distributed — cannot scale across multiple nodes.
  • OCR requires Poppler and Tesseract on the host machine or in the container (included in Dockerfile; manual install for local dev).
  • Query result caching is per-process and resets on restart.
  • The frontend does not yet have dedicated automated tests.
  • Static frontend serving depends on manually copying built assets from frontend/dist/ into static/.

18 · Key Lessons

What I learned building this in production

1. Retrieval quality is the system — not the LLM

The rerank threshold bug (RCA-9) proved this dramatically: gpt-4o-mini returned "information not available" for every query because the retriever dropped all chunks. A world-class LLM cannot produce a correct answer from empty context. Fix retrieval before tuning the generator.

2. Hybrid retrieval is worth the complexity for business corpora

Users ask both semantic questions ("what are the risks?") and exact-match questions ("what is clause 4.2?"). Dense-only retrieval handles the first; BM25 handles the second. The ensemble at 0.3/0.7 weights handles both — and the added complexity is contained entirely in _build_hybrid_retriever().

3. LangChain 0.3 API contracts require a boundary guard

create_retrieval_chain passes a dict in some code paths and a string in others, depending on chain composition. SafeQueryWrapper at the system boundary — once — is far better than adding isinstance checks in every inner retriever. The wrapper is a BaseRetriever, so LangChain's own dispatch logic handles extraction correctly.

4. Agentic routing belongs at the route layer, not inside retrieval

Compound splitting, adaptive MultiQuery, patient lookup routing, and injection guarding all live in app/routes/query.py — before retrieval begins. This keeps retrieval components simple and testable while routing logic evolves independently.

5. Deterministic structured lookup is not optional for tabular data

Semantic retrieval over row-chunked CSV data produces approximate matches. For patient-style queries ("list all female patients from Chennai"), exact row filtering via patient_lookup.py is the correct solution. The agentic routing layer detects the intent and bypasses vector retrieval entirely — zero hallucination risk for tabular lookups.

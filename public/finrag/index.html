01 · Project Overview

What FinRAG Engine v2 is and who it's for

FinRAG Engine v2 is a document-grounded question answering platform for teams that need fast, explainable answers from uploaded business files. It combines a FastAPI backend, a FAISS-based hybrid retrieval pipeline, and a React frontend for ingestion, monitoring, and query workflows.

The repository is suitable for: product demos that need visible source citations, internal knowledge assistants over PDFs and spreadsheets, multi-tenant document workspaces scoped by user_id, and structured CSV/XLSX lookups where exact row retrieval matters.

LLM model
gpt-4o-mini
OpenAI · tenacity retry · 45s timeout
Embedding model
text-embedding-3-small
1536 dims · indexing + reranking
Vector store
FAISS
faiss-cpu · persisted to disk · FileLock writes
API framework
FastAPI 0.111
Async · Pydantic v2 · Swagger in dev

Core capabilities

  • Ingests PDF, DOCX, XLSX, CSV, and TXT files
  • Stores embeddings in a local FAISS index under vector_store/
  • Uses hybrid retrieval: BM25 keyword search plus FAISS semantic search
  • Re-ranks retrieved chunks with an embedding similarity filter
  • Expands harder questions with MultiQuery retrieval when enabled
  • Splits compound questions into focused clauses before answering
  • Supports metadata filtering by user_id, doc_type, and source
  • Rejects obvious prompt-injection patterns at request validation time
  • Tracks ingestion jobs asynchronously and exposes job polling endpoints
  • Ships with a React frontend for authentication, document upload, dashboarding, and question answering

Demo-ready behaviours

  • Upload progress and ingestion status are visible in the UI
  • The dashboard shows document count, chunk count, vector count, recent queries, and latency posture
  • The query workspace shows staged loading, confidence, retrieval metadata, and expandable source cards
  • OCR dependency status is surfaced through GET /health and reflected in the frontend

02 · Why This Project Matters

The problem with most chat interfaces in demos

Most chat interfaces fail in demos because they answer confidently without showing evidence. FinRAG Engine v2 is built around the opposite behaviour.

Grounded
Answers stay grounded in uploaded documents
The LLM is instructed to answer only from retrieved context. If the answer isn't in the documents, it says so — it does not hallucinate.
Citations
Every factual answer is backed by citations
Responses include [source: filename.pdf, page_4] inline. The frontend shows expandable source cards with content snippets and chunk IDs.
Immediate
Ingestion is immediate — no retraining required
New uploads can be queried without retraining. The vector store is updated and the pipeline cache is invalidated on completion.
Transparent
The UI exposes confidence, latency, and source chunks
Instead of hiding retrieval, the query workspace shows staged loading, confidence score, retrieval metadata, and expandable source cards.
Exact
Structured table queries use deterministic row matching
Patient-style CSV/XLSX queries bypass semantic retrieval entirely and use exact row filtering — zero hallucination risk for tabular lookups.
Why this matters for demos

This makes the project useful both as an engineering foundation and as a demo artifact that explains itself clearly to customers, operators, and investors. The UI's transparency is the product signal that distinguishes it from generic chat interfaces.

04 · Retrieval & Answer Flow

Step-by-step: from upload to grounded answer

1
Document uploaded through POST /api/ingest
The backend validates extension and size, computes SHA256, checks dedup state, and saves the file under data/raw/. Returns a job_id immediately (HTTP 202).
2
Background ingestion task loads the file
A format-specific loader is dispatched by extension via the factory pattern: pdf_loader, docx_loader, excel_loader, txt_loader. All blocking work runs in asyncio.to_thread().
3
Format-aware chunking
The file is chunked with format-aware defaults: PDF=1200/200, DOCX=800/150, TXT=1000/150, CSV/Excel=1500/0 using RecursiveCharacterTextSplitter.
4
Metadata enrichment on every chunk
Each chunk receives: source, page_or_sheet, doc_type, chunk_id (shortuuid), ingested_at (ISO 8601 UTC), file_hash (SHA256), and user_id.
5
Embed and persist to FAISS
Chunks are embedded with OpenAI text-embedding-3-small in batches of 200 and written to FAISS with filelock.FileLock. Checkpointed after every batch.
6
Query normalisation and routing
A query sent to POST /api/ask is normalised, checked against the query result cache (TTL=300s), optionally decomposed into focused clauses, and checked for structured patient-table routing.
7
Retrieval: hybrid search + reranking + MultiQuery
The retriever combines BM25 keyword search (0.3) and FAISS semantic search (0.7), re-ranks candidates by cosine similarity, and optionally generates 2 alternate phrasings for harder questions.
8
Grounded generation
The generator answers using only the provided context and returns: answer, sources, confidence score, retrieval metadata, and request ID. Hallucination is prevented by the "answer ONLY from context" system prompt.
05 · Chunking Strategy

Why chunking is the most underestimated decision in RAG

Chunking directly determines retrieval quality. Chunks too large lose precision. Chunks too small lose context. The optimal size is format-dependent: a dense financial PDF needs larger windows than a structured CSV row.

python · src/chunking.py
# Per-format defaults: (chunk_size, chunk_overlap)
_FORMAT_DEFAULTS = {
    "pdf":   (1200, 200),  # dense financial text
    "txt":   (1000, 150),  # paragraphs pre-split by loader
    "docx":  (800,  150),  # section-grouped; keep sections coherent
    "csv":   (1500, 0),    # rows are independent; no overlap needed
    "excel": (1500, 0),
}

06 · Embeddings & Vector Store

Converting text to vectors and storing them safely

Each chunk is converted to a 1536-dimensional vector by text-embedding-3-small. The embedding model at indexing time must match at query time — swapping models requires full re-indexing.

python · src/embeddings.py
EMBED_BATCH_SIZE = 200   # safe batch — under OpenAI 300k token/req limit
_LOCK_TIMEOUT    = 60    # seconds to wait for FileLock before raising

with filelock.FileLock(lock_path, timeout=_LOCK_TIMEOUT):
    for idx, batch in enumerate(batches):
        if vectorstore is None:
            vectorstore = FAISS.from_documents(batch, embeddings)
        else:
            vectorstore.add_documents(batch)
        vectorstore.save_local(persist_path)  # checkpoint after each batch
C2 — Race condition fix

Multiple concurrent uploads writing to the same FAISS index files caused corruption. threading.Lock only protects within one process. Fix: filelock.FileLock at OS level — protects across processes and Docker containers sharing a volume.

SHA256 deduplication

Before saving any uploaded file, its SHA256 hash is checked against vector_store/ingested_files.json. Duplicate uploads return status: "skipped" immediately — no disk write, no embedding cost, no duplicate chunks in the index.

File-based persistence

The current implementation persists: uploaded files in data/raw/, FAISS artifacts in vector_store/, the dedup registry in vector_store/ingested_files.json, and write coordination in vector_store/.write.lock. This keeps the stack easy to run locally without adding external infrastructure.

07 · Hybrid Search Design

Why dense-only retrieval isn't enough for business corpora

The retrieval stack combines BM25 keyword retrieval for exact terms, identifiers, and entity matches; FAISS semantic retrieval using MMR for relevance plus diversity; and an embedding-based reranker to trim low-value candidates. This is a better fit than vector-only retrieval because users often ask both semantic and exact-match questions.

python · src/retriever.py
all_docs  = _extract_docs_from_faiss(vectorstore)
bm25_docs = _filter_docs(all_docs, filter_kwargs)  # tenant isolation
bm25_retriever = BM25Retriever.from_documents(bm25_docs)
ensemble = EnsembleRetriever(
    retrievers=[bm25_retriever, faiss_retriever],
    weights=[0.3, 0.7],  # BM25 / FAISS via Reciprocal Rank Fusion
)
Multi-tenancy

Documents and queries can be scoped by user_id. Filtering is applied both at retrieval time (FAISS native metadata filter + BM25 pre-filter) and in document listing, so multiple user workspaces can coexist on the same service instance without data leakage.

08 · Retrieval Pipeline

Five layers applied in order on every query

1
BM25 + FAISS/MMR via EnsembleRetriever
Sparse keyword (BM25 weight=0.3) + dense semantic (FAISS/MMR weight=0.7). MMR uses lambda_mult=0.6 for relevance-diversity balance — prevents near-identical chunks dominating the results.
2
_SafeEmbeddingsFilter — re-ranking with safety net
Recomputes cosine similarity between query embedding and every candidate. Drops chunks below RERANK_THRESHOLD=0.30. Safety net: if ALL candidates would be dropped, top-1 is always returned — the pipeline never delivers empty context to the LLM.
3
MultiQueryRetriever — optional query expansion
Generates MULTI_QUERY_NUM_QUERIES=2 alternative phrasings using a cached gpt-4o-mini singleton. Retrieves for all variants and returns the union. Adaptively disabled for short queries (≤4 words, <30 chars).
4
SafeQueryWrapper — system boundary input guard
Outermost BaseRetriever. LangChain's create_retrieval_chain may pass {"input": str} instead of a plain string. This wrapper normalises all input — making all inner components immune to LangChain API changes (fixes RCA-5).
5
Compound Query Orchestration
Multi-part questions split into focused clauses. Each clause retrieves independently using COMPOUND_QUERY_TOP_K=4 chunks. All clause contexts feed one LLM call with a structured COMPOUND_PROMPT.
RCA-9 — The threshold bug that killed all retrieval

The original RERANK_THRESHOLD=0.75 dropped every chunk for text-embedding-3-small. This model produces cosine similarities of 0.30–0.65 for relevant content — never approaching 0.75. Every query returned "This information is not available." Fix: lower to 0.30 + safety-net fallback returning top-1 when all candidates fall below threshold.

09 · Generation & Prompts

Building grounded answers from retrieved context

python · src/generator.py — system prompt
SYSTEM_PROMPT = """You are FinRAG, an expert document assistant.

Answer the user's question using ONLY the provided context passages.
Each passage starts with a [File: X | Location: Y] label for citations.

Rules:
1. Be concise and direct.
2. Use only supported facts from the context.
3. Cite every factual claim as [source: <filename>, <location>].
4. If the context does not contain the answer, say exactly:
   This information is not available in the provided documents.
5. Preserve exact figures, names, and terminology from the source.

Context passages:
{context}"""

Rule 4 is the hallucination guard. The DOCUMENT_PROMPT template embeds [File: X | Location: Y] into every passage before it reaches the LLM — citations always reference real metadata. For compound queries, a separate COMPOUND_PROMPT instructs the LLM to answer each clause in its own ### heading using only the context attached to that specific clause.

All LLM calls use tenacity: 4 attempts, exponential backoff (2s min, 30s max), retrying on any exception. Timeout is 45 seconds. max_tokens is capped at 700 (single question) or 900 (compound) to control cost and latency.

10 · Agentic Patterns

Where the system makes intelligent routing decisions

Adaptive
Adaptive MultiQuery activation
Detects query complexity before invoking MultiQueryRetriever. Short queries (≤4 words, <30 chars) skip it — reducing cost. Controlled by _is_complex_query() in app/routes/query.py.
Compound
Multi-intent clause splitting
Queries with 2+ intent starters split into focused clauses. Each clause retrieves independently — prevents blended retrieval across unrelated topics (fixes RCA-12).
Structured
Deterministic patient table lookup
When a question looks like a patient-table request, the backend bypasses vector retrieval and scans uploaded CSV/XLSX rows directly using PatientQueryFilters. Exact row matches; zero hallucination risk.
Failsafe
Injection guard + graceful failsafe
Prompt injection patterns blocked at QueryRequest validation before any retrieval occurs. If retrieval returns no context, returns "No relevant context found" — never hallucinating from empty context.

16 · Bug Register

Critical issues found and fixed across 21 phases

Critical — production correctness

C1
Blocking I/O in async route handlers
CPU-bound ingestion ran synchronously inside async def handlers, blocking the event loop. All concurrent requests queued behind one slow upload.
→ Fix: asyncio.to_thread() for all blocking ops in _run_ingestion()
C2
FAISS race condition — concurrent ingestion corrupted the index
Multiple uploads writing to the same FAISS files simultaneously. threading.Lock doesn't protect across processes or Docker containers.
→ Fix: filelock.FileLock at OS level on vector_store/.write.lock (60s timeout)
C3
Cache key missing top_k — wrong pipeline served silently
A query with top_k=4 would load the pipeline built for top_k=8. Wrong chunk count, wrong results — silently, every time.
→ Fix: cache key = f"topk={top_k}|filters={...}|mq={use_multi_query}"
C4
Filename collision — second upload overwrote the first
Two users uploading report.pdf resulted in the second overwriting the first on disk.
→ Fix: UUID-prefixed filenames: {uuid4().hex[:8]}_{original_filename}
C6
File size check after full read — OOM risk
The size guard ran after the entire file was buffered. A 500MB upload exhausted RAM before rejection triggered.
→ Fix: 64KB streaming chunks with early reject before buffering completes

RAG pipeline — retrieval correctness

RCA-4
_SafeEmbeddingsFilter was a plain class — reranking bypassed entirely
LangChain's MultiQueryRetriever calls .invoke() on the wrapped retriever. A plain class has no invoke method — __getattr__ delegation skipped reranking. Every query got unranked results.
→ Fix: converted to proper BaseRetriever subclass — .invoke() routes through _get_relevant_documents()
RCA-5
Missing SafeQueryWrapper — dict input crashed inner retrievers
LangChain's create_retrieval_chain passes {"input": str} in some code paths. BM25 and FAISS expect plain str — TypeErrors at retrieval time.
→ Fix: SafeQueryWrapper(BaseRetriever) as mandatory outermost layer; normalises dict → str
RCA-7
MultiQueryRetriever num_queries silently ignored — always used 3
Setting retriever.num_queries = 2 post-construction raises ValueError in Pydantic v2. Assignment silently dropped. All queries used 3 variants regardless of settings.
→ Fix: custom PromptTemplate passed to from_llm(prompt=...) embedding count in template text
RCA-9
Rerank threshold 0.75 dropped ALL chunks — empty context on every query
text-embedding-3-small produces cosine similarities of 0.30–0.65 for relevant content. The 0.75 threshold was above the model's maximum. Every query returned "This information is not available."
→ Fix: threshold lowered to 0.30 + safety-net returning top-1 when all candidates fall below
RCA-12
Mixed-intent queries retrieved blended context — both topics answered poorly
A compound query caused single retrieval over both topics simultaneously. Top-k chunks mixed content from both domains, diluting both answers.
→ Fix: clause splitting → independent retrieval per clause → single compound generation pass
H4
Misleading health check — healthy on empty or corrupt index
GET /health checked directory existence only. Empty vector_store/ returned vectorstore_loaded: true.
→ Fix: faiss.read_index().ntotal — true only when ntotal > 0

19 · Limitations & Next Steps

Current limits and the path to production scale

This codebase is strong for local use, demos, and single-node deployments, but there are important scaling limits:

  • pipeline_cache, query_result_cache, and job_registry are in-memory and not shared across multiple backend workers
  • FAISS is stored on local disk and is not horizontally distributed
  • OCR requires Poppler and Tesseract on the host machine or in the container (included in Dockerfile)
  • Query result caching is per-process and resets on restart
  • The frontend does not yet have dedicated automated tests
  • Static frontend serving depends on manually copying built assets from frontend/dist/ into static/

Recommended next production steps

Redis
Replace in-memory caches and job state with Redis
All three caches are imported from a single app/state.py module — making the migration straightforward without touching route logic.
Queue
Move background ingestion to a dedicated queue
Celery or a similar queue if throughput grows. The current asyncio.to_thread() approach has limits at high ingestion volume.
Vector DB
Add a managed vector store for multi-node scale
Replace src/embeddings.py with Pinecone, Weaviate, or Qdrant for horizontal distribution. The rest of the pipeline is vector-store-agnostic.
Tests
Add frontend integration or E2E tests
Playwright or Cypress end-to-end tests covering the upload → ingest → query → citation workflow. Add a repository license file before wider public distribution.

22 · Key Lessons

What I learned building this in production

1. Retrieval quality is the system — not the LLM

RCA-9 proved this: gpt-4o-mini returned "information not available" for every query because the retriever dropped all chunks. A world-class LLM cannot produce a correct answer from empty context. Fix retrieval before tuning the generator.

2. Hybrid retrieval is worth the complexity for business corpora

Users ask both semantic questions ("what are the risks?") and exact-match questions ("what is clause 4.2?"). The 0.3/0.7 ensemble handles both — and the added complexity is contained entirely in _build_hybrid_retriever().

3. LangChain 0.3 API contracts require a boundary guard

create_retrieval_chain passes a dict in some code paths and a string in others. SafeQueryWrapper at the system boundary — once — is far better than isinstance checks in every inner retriever.

4. Agentic routing belongs at the route layer, not inside retrieval

Compound splitting, adaptive MultiQuery, patient lookup routing, and injection guarding all live in app/routes/query.py — before retrieval begins. Retrieval components stay simple, testable, and reusable while routing logic evolves independently.

5. Deterministic structured lookup is not optional for tabular data

Semantic retrieval over row-chunked CSV produces approximate matches. For patient-style queries, exact row filtering via patient_lookup.py is the correct solution — zero hallucination risk for tabular lookups.

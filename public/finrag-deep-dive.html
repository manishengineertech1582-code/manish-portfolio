1Project Overview

FinRAG Engine is a production-grade Retrieval-Augmented Generation (RAG) system for intelligent question answering over PDF documents. It represents a complete implementation of the AI engineering stack — specifically the Month 3 milestone of building a serious, end-to-end RAG application with a real interface, evaluation metrics, and production deployment.

The system indexes four technical PDFs — the original Transformer paper (Attention Is All You Need), Hands-On Large Language Models, Fundamentals of Deep Learning, and 6GHz Spectrum Details — and serves natural language answers through a Claude-style dark chat interface.

💡 Production Mindset
Projects running only on your laptop create zero business value. FinRAG Engine addresses this directly — packaged with a Dockerfile, health check endpoint, non-root container execution, and uvicorn workers for production deployment.

2Tokenization, Context Windows & Temperature
2.1 Tokenization at Indexing Time

The chunking strategy in src/chunking.py is calibrated specifically around tokenization:

ParameterValueTokenization Rationale
DEFAULT_CHUNK_SIZE800 characters~200 tokens — within embedding model's optimal range
DEFAULT_CHUNK_OVERLAP150 characters~37 tokens — preserves sentence context at boundaries
SplitterRecursiveCharacterTextSplitterSplits by paragraphs then sentences — preserves semantic units
2.2 Token Cost Breakdown Per Query

Every call to gpt-4o-mini involves four token-priced components:

ComponentTokens (approx)Cost @ gpt-4o-mini
System prompt (SYSTEM_PROMPT)~110$0.0000165
Retrieved context (k=8 chunks × ~200 tokens)~1,600$0.000240
User question~10–20$0.0000023
LLM answer (output)~300–500$0.000240
Total per query~2,030~$0.000479
⚠️ Token costs scale fast
A prompt using 2,000 tokens when 500 would work is 4× the cost at scale. FinRAG's k=8 retrieval window is intentionally sized — enough to cover multi-topic compound questions without blowing the context budget.
2.3 Context Window

gpt-4o-mini has a 128,000 token context window. FinRAG's pipeline is designed so the total prompt never exceeds ~3,000 tokens under normal operation. The MultiQueryRetriever decomposes compound questions into sub-queries, each retrieving k=8 chunks. After deduplication, merged context reaches 1,600–2,400 tokens — well within budget.

2.4 Temperature
TemperatureBehaviourFinRAG Usage
0.0 (default)Fully deterministic, highest factual precision✅ Answer generation + query decomposition
0.0–0.3Low randomness, consistent outputsRecommended for factual RAG
0.7–1.0Creative, varied outputs❌ Not suitable for document QA

Setting temperature=0.0 ensures deterministic answers — critical for evaluation. You need reproducible outputs to measure whether retrieval changes actually improve answer quality.

4Chunking Strategy — Deep Dive

Most RAG systems fail because retrieval is weak, not because the LLM is bad. Retrieval quality is directly determined by chunking strategy.

4.1 Strategy Comparison
StrategyProsConsFinRAG?
Fixed character sizeSimple, predictable, fastMay split mid-sentence✅ YES — 800 chars
Fixed token countPrecise token budgetingRequires tokenizer per split❌ Not used
Paragraph-basedPerfect semantic unitsHuge variance in chunk sizes❌ Not used
Sentence-basedClean unitsVery short chunks, low recall❌ Not used
Semantic chunkingHighest qualitySlow, expensive, complex🔮 Future improvement
4.2 Why 150-Character Overlap Matters

Consider a sentence spanning two chunk boundaries: "The attention mechanism uses queries, keys, and [BOUNDARY] values to compute weighted sums." Without overlap, neither chunk contains the complete sentence. With 150 characters of overlap, both chunks contain the boundary text — retrieval works regardless of which chunk FAISS returns.

src/chunking.py
DEFAULT_CHUNK_SIZE    = 800   # ~200 tokens — embedding model's optimal range
DEFAULT_CHUNK_OVERLAP = 150   # ~37 tokens — preserves sentence boundaries

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=150,
    # Split hierarchy: paragraphs → sentences → words → characters
)
chunks = splitter.split_documents(documents)
# 4 PDFs → 2,485 chunks
4.3 Metadata Preserved on Every Chunk

Every chunk retains its full lineage: source (PDF file path), page (0-based page number), and chunk_id (sequential index). This flows all the way to the API response — the chat UI displays p.6 Transformer-attention-is-all-you-need-Paper.pdf for each source citation.

5.2 BUG-16 — Wrong Embedding Model (Critical)
🔴 BUG-16: Index built with model A, searched with model B
The original pipeline.py used OpenAIEmbeddings() with no model argument, defaulting to text-embedding-ada-002. But create_index.py used text-embedding-3-small. Completely different vector spaces — garbage similarity scores even when the vector store was healthy.
src/pipeline.py — fix
# BROKEN — defaults to ada-002, wrong vector space
embeddings = OpenAIEmbeddings()

# FIX — named constant, same model in embeddings.py AND pipeline.py
EMBEDDING_MODEL = "text-embedding-3-small"
embeddings = OpenAIEmbeddings(model=EMBEDDING_MODEL)
🔧 Rule: Pin ALL model names as named constants
BUG-16 would never have occurred if EMBEDDING_MODEL was defined as a constant on day one. Any file that uses a model name — embeddings.py, pipeline.py, generator.py — must reference the same constant.
6BUG-15 — The Pickle Serialization Bug

This is the most instructive bug in the project and the one most likely to come up in technical technical discussions. The symptom was clear: "This information is not available in the provided documents" — even for questions directly answered by the indexed PDFs.

FAISS similarity search was returning the correct page numbers (p.6 from the Transformer paper, p.255 from Hands-On LLM) — so the vectors were fine. But the answers were empty.

🔴 Root Cause: page_content appeared exactly 1 time across 2,485 documents
Python's pickle memo-optimization saw 2,485 Document objects with identical class structure and collapsed them into a single shared memo reference. On deserialization, 2,484 objects got empty default instances. The LLM received blank context and correctly answered "not available."
Diagnosis
Diagnosis command
python -c "
raw = open('vector_store/index.pkl', 'rb').read()
print(raw.count(b'page_content'))
# Healthy: 2485    Broken: 1
"
Root Cause — Python Pickle Memo-Optimization

The original embeddings.py used a batching loop with FAISS.add_documents(). This method stores Document objects by reference. When pickle serialized the docstore, it detected all 2,485 Document objects had the same class structure and wrote the full object once, using memo pointers for the rest. On deserialization, the 2,484 memo-referenced objects got empty default instances.

Before (broken) vs After (fixed)
# BROKEN — batching loop triggers pickle memo-optimization
vectorstore = FAISS.from_documents(batch_1, embeddings)
vectorstore.add_documents(batch_2)   # stores by reference
vectorstore.add_documents(batch_3)   # stores by reference
# Result: index.pkl has page_content exactly 1 time

# FIX — single call forces deep-copy semantics
vectorstore = FAISS.from_documents(docs, embeddings)
# LangChain handles API rate-limit batching internally
# Result: index.pkl has page_content 2,485 times
🔧 Fix: one line
Replace the entire batching loop with a single FAISS.from_documents(docs, embeddings) call. This forces deep-copy semantics — each Document gets its own independent object reference. Verify with the diagnosis command above — should output 2485.

8Retrieval Strategies

The most common reason RAG systems fail is weak retrieval, not a weak LLM. If retrieved context does not contain the answer, the LLM cannot generate it — regardless of capability.

8.1 MultiQueryRetriever — Compound Question Fix

Semantic search fails for compound questions like "What frequency bands are covered in 6GHz spectrum AND what is the difference between supervised learning and LLMs?" The combined query embedding clusters around the dominant topic — the minority topic gets zero coverage.

The fix: MultiQueryRetriever uses gpt-4o-mini to decompose the question into sub-queries. For the example above it generates:

  • "What frequency bands are in the 6GHz spectrum?"
  • "What is supervised learning?"
  • "What is deep learning?"
  • "What are large language models?"

Each sub-query gets its own FAISS search with k=8. Results are merged and deduplicated. The LLM receives chunks from all relevant documents simultaneously. This adds one LLM call (~$0.00012) but dramatically improves recall for complex questions.

src/retriever.py
from langchain.retrievers.multi_query import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=base_retriever,    # FAISS k=8
    llm=ChatOpenAI(model="gpt-4o-mini", temperature=0),
)
# One compound question → 3-4 sub-queries → merged, deduplicated results

9Generation — LangChain 0.3+ Chain API
9.1 Migration from Deprecated API

The deprecated RetrievalQA.from_chain_type contained a critical bug: when combined with a custom prompt, the StuffDocumentsChain silently dropped context injection — causing the LLM to answer from training memory rather than retrieved documents.

src/generator.py — old vs new
# OLD — deprecated, context silently dropped with custom prompts (BUG-13)
chain = RetrievalQA.from_chain_type(llm, chain_type_kwargs={"prompt": PROMPT})

# NEW — LangChain 0.3+ standard, context always injected correctly
combine_docs_chain = create_stuff_documents_chain(llm, PROMPT)
qa_chain           = create_retrieval_chain(retriever, combine_docs_chain)

# CRITICAL: Key names changed — silent failures if wrong
#   Input:   {"query": q}           →  {"input": q}
#   Answer:  result["result"]       →  result["answer"]
#   Sources: result["source_documents"] →  result["context"]

10Evaluation — Hit@K & MRR

Most RAG projects have no idea if their system actually works well. Evaluation is what separates a production system from a demo. FinRAG implements two standard Information Retrieval metrics in src/evaluation.py.

MetricFormulaBest ScoreWorst ScoreWhat It Measures
Hit@K1 if correct doc in top-K else 01.00.0Retrieval presence — did we surface the answer at all?
MRR1 / rank of first correct doc1.0 (rank 1)0.0 (not found)Retrieval rank — is the right doc at the top?
src/evaluation.py — usage
hit = hit_at_k(retrieved_docs, ground_truth_doc_id=5)
# Returns 1 if a chunk from page 5 is in top-K results

mrr = mean_reciprocal_rank(retrieved_docs, ground_truth_doc_id=5)
# Returns 1.0 if page 5 chunk is rank 1, 0.33 if rank 3

# Document ID priority resolution:
# id → chunk_id → page → source (file path)
💡 BUG-15 would have been caught immediately
If Hit@K and MRR were running from day one, BUG-15 would have shown 0.0 across the board — immediately flagging that retrieval was broken. Instead it required binary inspection of index.pkl. Write evaluation before writing the pipeline.
11Agents vs Chains

FinRAG deliberately uses a deterministic retrieval chain rather than an agent. Understanding why reveals important production engineering judgment.

PropertyAgent ArchitectureFinRAG Chain
Cost10+ LLM calls per query2 LLM calls (decompose + generate)
Latency10–30 seconds typical~2–4 seconds total
PredictabilitySame input → different pathsDeterministic at temperature=0
ObservabilityComplex trace requiredSimple request/response logging
Error surfaceAgent can go down wrong pathsMinimal — fixed pipeline
💡 Simpler agents often outperform complex ones
FinRAG's MultiQueryRetriever is the simplest possible agent: one LLM call to decompose the question, then parallel deterministic searches. It adds ~$0.00012 but dramatically improves recall for compound questions. Maximum benefit, minimal cost and complexity.

15What I Would Improve
Retrieval
  • Hybrid Search — combine semantic + BM25 keyword matching (rank-bm25 already in requirements.txt). BM25 catches exact keyword matches that semantic search misses — critical for technical terms and model names.
  • Re-ranking — add cross-encoder second-pass (sentence-transformers in requirements.txt). Retrieves 20–30 candidates with bi-encoder, re-ranks for precision.
  • Semantic chunking — split at topic boundaries rather than character counts.
  • Chunk size tuning — vary 400–1200 chars, measure Hit@K and MRR on a ground truth test set, pick the optimal size.
Production
  • Streaming responses — user sees words within 500ms instead of waiting 3 seconds for a complete answer.
  • Conversation memory — follow-up questions like "can you elaborate?" currently fail. Add conversation buffer window memory.
  • Observability — LangSmith or Langfuse for per-query traces, token counts, and chunk inspection.
  • Caching — Redis cache with query embedding as key. Repeat questions cost $0.00 and return near-instantly.
  • Cost tracking — server-side per-request cost logging enables per-user analysis and budget alerts.
Evaluation
  • Ground truth test set — 50–100 questions with known source pages. Run Hit@K and MRR automatically on every code change (CI evaluation pipeline).
  • LLM-as-judge — second LLM call scores answer quality against a reference answer.
  • MLflow integration — mlflow==2.22.1 already in requirements.txt. Wire evaluation metrics into experiment tracking to compare chunking strategies over time.
16Key Lessons & Reflection
Why this problem?

Technical documentation contains answers — but they are buried in PDFs that take hours to read. A RAG system reduces "I have a question" to "I have an answer" from hours to seconds, with full source attribution so every claim is verifiable.

Why RAG over fine-tuning?

The knowledge base changes — new PDFs are added by dropping them in data/ and rerunning create_index.py. Fine-tuning costs thousands of dollars and gives no source attribution. RAG with gpt-4o-mini costs ~$0.000479 per query. RAG answers are fully auditable.

Key trade-offs made
DecisionChoice MadeTrade-off Accepted
Vector DBFAISS (local)No distributed search, no real-time index updates without full rebuild
LLMgpt-4o-mini10× cheaper than gpt-4o, slightly lower reasoning on very complex multi-hop queries
ChunkingCharacter-basedSome chunks split mid-sentence — mitigated by 150-char overlap
PipelineIn-processVector store held in FastAPI memory — acceptable at 2,485 chunk scale
What I would do differently
  1. Write evaluation before the pipeline. Hit@K = 0 would have caught BUG-15 immediately instead of requiring binary inspection of index.pkl.
  2. Pin ALL model names as named constants. BUG-16 would never have occurred.
  3. Build streaming responses from day one. Retrofitting streaming into a completed pipeline is harder than building it in from the start.
💡 This field rewards builders over studiers
The ability to diagnose BUG-15 as a Python pickle memo-optimization issue requires understanding how object serialization works at the byte level. That is not something you learn from tutorials. That is what you learn from building — and shipping to production. Every project teaches you things that reading never will.
